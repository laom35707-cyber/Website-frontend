一、你对 Wiki 的核心要求

整体定位
你希望它更接近 Wikipedia，而不是简单的 Markdown 预览器。目标是功能完整、结构清晰、可扩展的知识系统。

前端目标

三栏结构（左导航 / 中编辑 / 右预览+目录）

更专业的 UI 风格（接近 Wikipedia 而不是博客）

自动目录（TOC）

内部跳转

搜索功能

单页加载体验（无需刷新）

自动保存或本地草稿能力

文档支持能力
你明确要求：

支持 Markdown 文件上传并直接解析

支持 PDF 文件上传并渲染

支持 Google Drive 链接自动识别

支持 OneDrive 链接自动解析

媒体嵌入能力（视频、图片）

自定义跳转语法（关键定制要求）

你定义了特殊语法：

"a"{b}


规则是：

页面只显示 a

点击跳转到标题完全等于 b 的页面

区分大小写

必须完全匹配

{} 不显示

自动识别并解析

这本质上是一个“精确标题跳转协议”。

解析行为要求

自动识别文档中的跳转语法

自动识别 Google Drive

自动识别 OneDrive

自动隐藏内部标记符号

Markdown 与自定义语法并存

二、当前系统的实现现状

UI 架构现状

已实现：

三栏布局

搜索框

页面列表

Markdown 编辑器

预览区

自动 TOC

内部链接处理

SPA 风格加载

不足：

还不是 Wikipedia 等级视觉规范

没有历史版本记录

没有分类/标签系统

没有引用系统（<ref>）

没有讨论页

没有页面历史对比

没有侧边分类树

文件与媒体支持现状

已实现：

Markdown 文件导入

PDF 文件上传

PDF.js 渲染第一页

Google Drive 自动转 preview

OneDrive embed

图片嵌入

限制：

PDF 目前只渲染第一页

没有文件统一存储管理

没有文件列表

没有文件权限

跳转系统现状

已实现：

"a"{b} 精确匹配跳转

区分大小写

必须完全匹配

如果不存在会提示

限制：

每次搜索会加载全部页面索引

页面数量大时性能下降

没有缓存层优化

搜索系统现状

已实现：

标题匹配

内容匹配

前端全文遍历

限制：

无权重

无索引优化

数据量大时会变慢

无模糊拼写纠正

Markdown 解析能力

当前为轻量级手写解析：

支持：

标题

加粗

斜体

图片

链接

PDF embed

自定义语法

不足：

无代码高亮

无表格支持

无任务列表

无脚注

无数学公式

无 GitHub 风格扩展
